# PR Plan: JACLibc Upstream Contributions

## Overview

We're splitting our monolithic fork changes into focused, small PRs for `frinknet/JACLibc`.
Our fork stays intact for our own work. Each PR is a clean branch off `upstream/main`.

### Guiding Principles
- **Frink's word is final** — if he wants it done differently, we adapt
- **Every PR needs clear justification** — why each change is necessary
- **Never edit `x/*.h` directly** — those are auto-generated by awk scripts in `utils/`
- **No duplicate helpers** — frink: "You shouldn't have two helpers that do the same thing with different return types"
- **No JSRUN/WASM changes** — intentionally stubbed, not ready yet
- **No math changes** — frink is protective of math code, needs strong justification

## PR 1: Darwin ARM64 Compilation Fixes

**Branch:** `pr/darwin-arm64-compilation`
**Priority:** Highest — this is what frink explicitly asked for
**Justification:** Every change in this PR is the minimum required to make JACLibc compile
and link on macOS Apple Silicon. Without any one of these, compilation or linking fails.

### Changes

#### `include/arch/arm64.h`
1. **Darwin `_start` entry point** — Add Darwin-specific `_start` that calls `__start_main`
   (with underscore prefix) instead of `_start_main`. Mach-O requires the underscore prefix
   for C symbol linkage. Wrapped in `#if JACL_OS_DARWIN` / `#else` to preserve Linux behavior.

2. **Fix `#ifdef` syntax bug** — Change `#ifdef __ARCH_CLONE && JACL_OS_LINUX` to
   `#if defined(__ARCH_CLONE) && JACL_OS_LINUX`. The `#ifdef` preprocessor directive only
   takes a single token; the `&& JACL_OS_LINUX` part was silently ignored, meaning
   Linux-only clone code would run on Darwin and cause errors.

#### `include/config.h`
3. **Fix Mach-O init section format** — Change `JACL_INIT_SECTION` from `"__mod_init_func"`
   to `"__DATA,__mod_init_func"`. Mach-O section attributes require `segment,section` format.
   Without the `__DATA` segment prefix, the compiler rejects the section attribute on Darwin.

#### `include/fmt/macho.h`
4. **Null out init function pointers** — Set `INIT_START` and `INIT_END` to `NULL`.
   On Darwin, `dyld` automatically calls constructors placed in `__DATA,__mod_init_func`
   before `_start_main` runs. Unlike ELF, Mach-O does not generate `__start_`/`__stop_`
   section boundary symbols. The weak extern declarations either resolve to NULL or cause
   linker errors depending on the toolchain.

#### `include/fcntl.h`
5. **Darwin posix_fadvise / posix_fallocate stubs** — Darwin has no `SYS_fadvise64` or
   `SYS_fallocate` syscalls. `posix_fadvise()` is a no-op (advisory only, returning 0).
   `posix_fallocate()` falls back to `ftruncate` (best-effort). Wrapped in
   `#if JACL_OS_DARWIN` / `#else` to preserve Linux/BSD behavior.

#### `include/pthread.h`
6. **Fix pthread_exit syscall** — Change `SYS_thread_terminate` (a Mach trap) to
   `SYS_bsdthread_terminate` with the correct 5-argument BSD calling convention.
   `SYS_thread_terminate` is not a BSD syscall and doesn't terminate the thread correctly.

#### `include/sched.h`
7. **Darwin sched_yield no-op** — Add `JACL_OS_DARWIN` check before the `JACL_HAS_POSIX`
   check in `sched_yield()`. Darwin is POSIX-compliant so `JACL_HAS_POSIX` is true, but
   Darwin has no `SYS_sched_yield` syscall. Without this, the code calls a non-existent
   syscall. The no-op return is safe — the scheduler handles preemption automatically.

#### `include/unistd.h`
8. **Darwin getcwd implementation** — Darwin has no `SYS_getcwd`. Implemented via
   `open(".", 0, 0)` + `fcntl(fd, F_GETPATH, buf)`, which is the standard Darwin approach
   for getting the current working directory path. Wrapped in `#if JACL_OS_DARWIN` / `#else`.

#### `include/core/stdlib.h`
9. **Remove pthread_yield_np** — The `JACL_OS_DARWIN` branch in `__jacl_lock_acquire` called
   `pthread_yield_np()`, a pthread-level function not available in a freestanding C library.
   Remove the Darwin-specific branch so it falls through to `sched_yield()`, which now
   handles Darwin correctly via the sched.h fix above.

### NOT Included
- No `x/` file changes (auto-generated)
- No atomics changes (ARM64-general, separate PR)
- No `__jacl_bin_safety_void` (general bug fix, separate PR)
- No `renameat()` (general addition, separate PR)
- No `snprintf` circular dependency fix (general, separate PR)

## PR 2: Fix ARM64 Atomic Instructions

**Branch:** `pr/arm64-atomics`
**Priority:** High — correctness fix for all ARM64 targets
**Justification:** ARM64 (ARMv8 AArch64) uses `LDXR/STXR` exclusive load/store instructions.
ARM32 (ARMv7) uses `LDREX/STREX`. These are completely different instruction encodings.
The current code groups them with `JACL_ARCH_ARM32 || JACL_ARCH_ARM64` and uses ARM32
instructions for both, which causes assembler errors on any ARM64 target.

### Changes

#### `include/stdatomic.h`
1. **Split ARM64 from ARM32** — Separate the combined `JACL_ARCH_ARM32 || JACL_ARCH_ARM64`
   section into two architecture-specific sections:
   - `JACL_ARCH_ARM64`: Uses `ldar`/`stlr` for acquire/release load/store,
     `ldxr`/`stxr` for exchange and compare-and-swap operations
   - `JACL_ARCH_ARM32`: Keeps existing `ldrexb`/`strexb`, `ldrexh`/`strexh`,
     `ldrex`/`strex` (unchanged)

This affects all ARM64 platforms (Linux, Darwin, FreeBSD), not just Darwin.

## PR 3: Fix BSD Syscall Generator (SYS_nosys Duplicates)

**Branch:** `pr/fix-syscall-generator`
**Priority:** High — prerequisite for clean Darwin compilation
**Justification:** The awk generator outputs multiple `X(SYS_nosys, N, nosys)` entries
because many Darwin syscalls are stubbed as `nosys` in the XNU kernel source. When the
X-macro expands to an enum, `SYS_nosys` gets defined multiple times with different
values — a hard compilation error. The fix must go in the generator script, not the
generated file.

### Changes

#### `utils/get_syscalls_bsd.awk`
1. **Filter nosys/enosys entries** — In the processing logic, skip entries where the
   function name is `nosys` or `enosys`. These are unimplemented syscall placeholders
   that serve no purpose in the X-macro output and cause enum redefinition errors.

#### `include/x/darwin_generic.h` (regenerated, not hand-edited)
2. **Regenerate** — Run `utils/get_syscalls.sh` after fixing the awk script.
   Commit the regenerated output alongside the script fix.

## PR 5: Fix Allocator Safety Macro Return Type

**Branch:** `pr/fix-bin-safety-return`
**Priority:** Medium
**Justification:** `__jacl_bin_safety` macro uses `return 0` which breaks in void functions
(`__jacl_bin_push`, `__jacl_bin_remove`). Per frink's explicit feedback, do NOT create a
duplicate helper. Instead, change the void functions to return `int`.

### Changes (Reworked per Frink's Feedback)

#### `include/core/stdlib.h`
1. **Remove `__jacl_bin_safety_void`** — Delete the duplicate macro entirely
2. **Change function signatures** — `__jacl_bin_push` and `__jacl_bin_remove` return `int`
   instead of `void` (0 on safety failure via existing macro, 1 on success)

This makes the original `__jacl_bin_safety` macro work uniformly across all call sites.
No callers currently check return values from push/remove, so this is safe.

## PR 6: Fix Circular snprintf Dependency in setenv

**Branch:** `pr/fix-setenv-circular-dep`
**Priority:** Low
**Justification:** In a header-only library, `stdlib.h` calling `snprintf()` from `stdio.h`
creates a circular include chain. The replacement `strcpy`/`strcat` is safe because the
buffer is pre-allocated to exact size (`strlen(name) + strlen(value) + 2`).

### Changes

#### `include/stdlib.h`
1. Replace `snprintf(str, len, "%s=%s", name, value)` with:
   ```c
   strcpy(str, name);
   strcat(str, "=");
   strcat(str, value);
   ```
